What is Haskell?

Haskell是一个惰性的、函数式的编程语言，由一群学者在19世纪80年代晚期创建。因为当时有许多惰性的函数式语言，每个人都有他们的选择，因此（跨编程语言的）思想难以交流。所以一撮人聚在一起，糅合当时存在的编程语言中的精华思想和一小部分新理念到一个新语言中。Haskell诞生了

所以，Haskell的特性究竟如何？它是:

Functional

没有一个精确又被广泛接受的对functional一词的定义，不过当我们提及Haskell是个functional language时，我们通常指这两件事:

  • 函数是一等公民(first-class)，这意味着函数可以像任何其他其他值一样被使用

    译者注: 比如作为变量被保存或作为参数被传递

  • Haskell的程序往往关注如何对表达式求值，而非执行指令

将这两点组合在一起，我们得到了编程的全新思维模式，接下来的学习过程中我们将花大部分时间来探索这种思维模式。

Pure

Haskell的表达式总是引用透明 (referentially transparent)的，引用透明意味着:

  • 不可变性。所有东西（变量，数据结构...）都是不可变的
  • 表达式永远没有副作用 (side effects)，比如更新全局变量或者打印/输出到屏幕
  • 用同样的参数调用同样的变量永远会得到同样的结果（即纯函数）

这听上去可不得了。变量又不可变，函数又不能有副作用，Haskell如何做那些传统语言能做的事呢？反正可以，先调整思维模式（特别是当你曾经学习过FOP或OOP时）然后好好上课。当你真正切换到这种思维模式时，你会发现世界变得不同了:

  • 等式推理与重构 (Equational reasoning and refactoring): 像你在代数课上所学，在Haskell中，永远可以用等式替换等式
  • 并行性 (Parallelism): 对表达式求值(evaluating)可以轻易地并行，只要它们保证不会互相影响
  • 头不痛了 (Fewer headaches): 易用但不受限制的副作用与跨语句作用(action-at-a-distance)会导致程序难以调试、维护以及分析(reason about)

Lazy

在Haskell中，表达式仅在需要其值时被求值，这个设计简单但具有深远影响，我们将在整个学期中体会它。其中一些影响是:

  • 通过定义函数来定义控制结构(control structure)非常容易
  • 可以定义并使用无限大的数据结构
  • 使得更具组合性的编程风格(compositional programming)成为可能（见之后的wholemeal programming）
  • （这是缺点）分析时间和空间使用率更加复杂了

Statically typed

所有的Haskell表达式都有类型，并且类型都在编译时被检查。包含类型错误的程序不会编译成功，因此减少了运行时错误的可能

Themes

贯穿整个课程，我们将关注三个主题

Types

静态类型系统看上去非常讨厌。实际上，在诸如C++或Java的语言中，静态类型确实很讨厌:( 但这不是静态类型系统本身的问题，只是因为C++和Java的类型系统的表达能力太弱。在学期中，我们将会仔细学习Haskell的类型系统，而它可以:

  • 帮助理清思维和表达程序结构

    写Haskell程序的第一步通常是写下所有类型。得益于Haskell类型系统的表现力，这个过程中没有平凡琐碎的步骤，只会帮助你更好地理清写程序的思路

  • 帮助代码文档化

    在类型系统中，一个函数的类型定义可以透露很多关于它要如何被使用的信息，哪怕你完全没有阅读关于这个函数的文档

  • 将运行时错误转化为编译时错误

    在运行程序前就修复错误比疯狂测试然后期望它能work要好得多。“如果编译成功，那它必定正确”在大多数时候是玩笑话（因为在类型正确的程序上仍然可能出现逻辑错误），但是这句话在Haskell中比在其他语言中更具现实性

Abstraction

“Don't Repeat Yourself”是编程中经常被提起的咒语(mantra)，又被称为“Abstraction Principle”，其关键在于不该存在任何重复: 每个想法、算法、数据都应该只在代码里出现一次。关注代码的相似结构并分析其共性(commonality)即为抽象(abstraction)

Haskell最擅长抽象: 包括参数多态、高阶函数与类型类(type classes)都能帮助减少重复。我们探索Haskell的旅程将会有大半时间在关注抽象

Wholemeal programming

另一个我们将要探索的主题是wholemeal programming。这是一段来自Ralf Hinze的话:

函数式语言最擅长wholemeal programming，这个术语由Geraint Jones提出。Wholemeal programming意味着关注全局(to think big): 操作整个列表而非一小段元素；开发解的空间而非单个解；想象整个图而非单条路径。Wholemeal的方法经常能为问题提供新的视角。它很好地补充了投影编程(projective programming)的思想: 先解决一个通用的问题，再将这个通用问题转化成更具体的问题以提取我们需要的信息。

一个例子，看下面这段C/Java-ish的伪代码:

int acc = 0;
for ( int i = 0; i < lst.length; i++ ) {
  acc = acc + 3 * lst[i];
}

这段代码遇到了一个Richard Bird提出的叫“indexities”的问题: 它太关注低级的、如何根据index迭代一个数组的细节。同时它也把两个我们能轻易区分开的操作混合到了一起: 将数组的每个元素乘3 与 对结果求和

而在Haskell中，我们只需要写

sum (map (3*) lst)

我们在这学期将先学会如何这样编程，再考虑为何Haskell要这样设计、以及这种设计如何实现

Arithmetic

尝试在GHCi中执行以下语句并检查求得的值

ex04 = 8.7 / 3.1

译者注: 在这里，mod是一个函数，mod a b意味着用a, b两个参数调用mod

我们可以用`来包裹mod这种二元函数，来使其变为可中缀调用的形式。同时注意，负数必须整个用括号包起来，这很丑，但没办法:(

下面这个语句会报错

badArith1 = i + n

理由是

Couldn't match expected type ‘Int’ with actual type ‘Integer’

Haskell只允许相同的数值类型被相加，它不会帮我们做隐式转换（译者注: 隐式转换虽然方便，但会导致隐蔽的难以察觉的bug），如果想这样做，必须通过以下途径显式转换:

  • fromIntegral 把任何整数(Int / Integer)类型转换成其他任何种类的数值类型(Int / Integer / Double / Float)
  • round, floor, ceiling 把浮点数转换成整数(Int / Integer)

接下来试试这一句

badArith2 = i / i

它同样会报错

No instance for (Fractional Int) arising from a use of ‘/’

这是因为/只做浮点数除法，如果想做整数除法，请使用div:

ex09 = i `div` i 	   -- same as ex09 = div i i
ex10 = 12 `div` 5    -- same as ex10 = div 12 5

如果你已经习惯了其他会对数值做隐式转换的语言，那Haskell的“死板”可能会让你难受。但是，我向你承认你会习惯的，并且随时间推移你会越发感谢它。隐式数值转换让我们对数值相关的代码的思维变得马虎

Boolean logic

如你期待，Boolean值可以用 &&, ||, not三个操作符，比如:

ex11 = True && False
ex12 = not (False || True)

常用的比较也是可以的，包括==, /=, <, >, <=, >=

ex13 = ('a' == 'a')
ex14 = (16 /= 3)
ex15 = (5 > 3) && ('p' <= 'q')
ex16 = "Haskell" > "C++"

Haskell同样有if表达式 (if-expression): if b then t else f会在b == True时返回t，否则返回f。这和传统的if语句 (if-statement) 有区别。表达式与语句的区别在于表达式必须返回一个值，而语句代表一段控制指令

同样对于上面那段代码，if语句的自然语言表述是: 如果b就做t，否则执行f，因此else块在需要执行的f为空时可以省略，代表什么都不做。二if表达式的自然语言表示则是: 如果b，这个表达式的值为t，否则值为f，因此else不可省略，因为否则这个表达式在b不满足时就没有任何值

理想的Haskell程序不经常使用if，而是用模式匹配(pattern-matching)和门卫(guards)代替，参见下一部分

Defining basic functions

Pattern Matching

我们可以实现一个输入正整数n，返回1..n的和的函数如下:

-- Compute the sum of the integers from 1 to n.
sumtorial :: Integer -> Integer
sumtorial 0 = 0
sumtorial n = n + sumtorial (n-1)

第一行是声明，告诉Haskell这是一个接受Integer输出Integer的函数，剩下的子句（每一行）会被从上到下检查（根据函数参数），直到找到第一个符合条件的子句

比如sumtorial 0会匹配第一个子句，因此直接返回0；而sumtorial 3没有匹配到第一个子句，因此会检查第二个子句，第二个子句没有任何匹配要求（比如要求输入是0），它只是将输入用变量n指示，即n = 3，然后它返回n + sumtorial (n-1)，即3 + sumtorial 2，进一步变成3 + 2 + sumtorial 1，3 + 2 + 1 + sumtorial 0及3 + 2 + 1 + 0。简单的递归:p

上面的选择基于函数入参是否满足某个模式(pattern)来做出，0是一个模式，意味着输入了0，n是一个模式，意味着输入了单个变量，从上到下挨个匹配模式，直到找到匹配的（在这里n能匹配任何模式，但因为从上到下执行，所以如果输入是0，0总是比它先被匹配）。因此这种语法被称为模式匹配

Guards

选择也可以基于Boolean表达式的值做出，这种语法被称为门卫:

hailstone :: Integer -> Integer
hailstone n
  | n `mod` 2 == 0 = n `div` 2
  | otherwise      = 3*n + 1

一个函数的子句（在这里即为hailstone n ...，它其实是一个模式匹配子句，且永远会被匹配，只是返回值由门卫决定而非直接给出）里可以有很多个门卫，以|开头，每个门卫的等号前是条件，等号后是返回值表达式。和模式匹配很像，区别只在于它的条件是Boolean值，而非一个模式，otherwise可以理解为永远为True

当我们调用hailstone 3时，首先根据模式匹配，3符合n这个pattern，因此n = 3并继续对门卫求值，首先查看第一个门卫条件 n mod 2 == 0，由于3 mod 2 \= 0，因此第一个门卫不被匹配。接下来查看第二个门卫，由于其是otherwise，因此总是被匹配，于是函数返回3*3 + 1 = 10

一个更复杂的例子如下，它结合了多个模式匹配和守卫:

foo :: Integer -> Integer
foo 0 = 16
foo 1 
  | "Haskell" > "C++" = 3
  | otherwise         = 4
foo n
  | n < 0            = 0
  | n `mod` 17 == 2  = -43
  | otherwise        = n + 3

foo (-3), foo 0, foo 1, foo 36, foo 38 的值分别是多少呢？

作为布尔表达式和守卫的总结，假设我们希望抽象出一个测试整数是否为偶数的函数，第一个尝试如下:

isEven :: Integer -> Bool
isEven n
	| n `mod` 2 == 0 = True
	| otherwise      = False

这个函数能符合要求，但太复杂了，你能找到原因吗？

译者注: 可以改成更简单的

isEven :: Integer -> Bool
isEven n = n `mod` 2 == 0

不过Haskell已经有判断奇偶数的函数odd和even了

Pairs

我们可以把东西打包在一起成为一个Pair，就像Python里的元组一样

p :: (Int, Char)
p = (3, 'x')

()符号包裹类型时，就像(Int, Char)，表示一个指定形状的Pair类型（也即(Int, Char)和Int, Char等一样也是单独的类型）；()包裹实际值的字面量时，就像(3, 'x')则表示一个(Int, Char)的Pair的具体取值

由于指定了形状的Pair是单独的类型，我们可以像对其他类型一样对Pair应用模式匹配，并且可以在模式匹配过程中直接将Pair的值提取出来绑定到指定变量:

sumPair :: (Int, Int) -> Int
sumPair (x,y) = x + y

译者注: 在模式匹配中提取Pair的值的语法有点类似Python或Javascript的解构赋值，如果你曾经接触过它们的话

// Javascript
const o = { a: 1, b: 2}
const { a, b } = o;
# Python
a, b = (1, 2)

Haskell还有三元组(triples)、四元组(quadruples)...，但你永远不应该使用他们。我们下周会讲更好的将三个及以上变量打包在一起的方法

Using functions, and multiple arguments

要用多个参数调用函数，只要将参数挨个排在函数名后面，用空格分隔，就像:

f :: Int -> Int -> Int -> Int
f x y z = x + y + z
ex17 = f 3 17 8              -- calling f with arguments 3, 17 and 8

上面的例子用3, 17, 8三个参数调用f。关注定义包含多个参数的函数类型的语法，形如Arg1Type -> Arg2Type -> ... -> ResultType，这看起来可能很奇怪，为什么全是箭头？难道不应该写成Int Int Int -> Int吗？

实际上，这种语法并不是瞎写的——它有非常深刻且优美的理由，就蕴含在Haskell看待函数的方式之中。我们将在接下来几周学习它，现在你只要记住我的话就好了

注意，函数调用比中缀操作符有更高的优先级，因此这样写

f 3 n+1 7

并不意味着用3, n+1, 7调用f，而是被解释为(f 3 n) + (1 7)

所以正确的写法是

f 3 (n+1) 7

Lists

列表 (Lists)是Haskell最基础的数据类型之一

nums, range, range2 :: [Integer]
nums = [1, 2, 3, 19]
range = [1..100]                 -- [1, 2, 3, ..., 100]
range2 = [2, 4..100]						 -- [2, 4, 6, ..., 100]

Haskell和Python一样支持列表推导式 (List Comprehension)，你可以在这找到更多的解释Haskell趣学指南 第二章 入门

如我刚才所说，字符串就是字符的列表，String是[Char]的缩写，并且字符串字面量的语法（用双引号包裹文本）也是Char列表的语法糖，在下面的代码中，hello1和hello2完全等价

-- hello1 and hello2 are exactly the same.

hello1 :: [Char]
hello1 = ['h', 'e', 'l', 'l', 'o']

hello2 :: String
hello2 = "hello"

helloSame = hello1 == hello2        -- it's True

这也意味着，所有能够处理普通列表的基础库都能处理字符串

Constructing lists

最简单的列表当属

emptyList = []

其他列表都是从空列表构建而来的，列表实际就是一堆元素的组合，利用cons操作符:，我们可以将一个元素拼接到一个列表上，构建出一个新的列表。被拼接的元素会出现在新列表的头部

ex18 = 1 : []
ex19 = 3 : (1 : [])
ex20 = 2 : 3 : 4 : []
ex21 = [2, 3, 4] == 2 : 3 : 4 : []  -- it's True

所以，[2,3,4]其实就是2:3:4:[]的语法糖，而且我们意识到，这里所说的列表其实是单向链表，而不是数组 (Array)

-- Generate the sequence of hailstone iterations from a starting number.
hailstoneSeq :: Integer -> [Integer]
hailstoneSeq 1 = [1]
hailstoneSeq n = n : hailstoneSeq (hailstone n)

让我们用调用hailstoneSeq的过程来代替对它的解释吧:

hailstoneSeq 3 
matching pattern n => hailstoneSeq 3 = 3 : hailstoneSeq (hailstone 3) 
								   => hailstoneSeq 3 = 3 : hailstoneSeq (10)
matching pattern n => hailstoneSeq 3 = 3 : (10 : hailstoneSeq (hailstone 10))
                   => hailstoneSeq 3 = 3 : (10 : hailstoneSeq 0)
matching pattern n => hailstoneSeq 3 = 3 : (10 : (0 : hailstoneSeq (hailstone 0)))
                   => hailstoneSeq 3 = 3 : (10 : (0 : hailstoneSeq 1))
matching pattern 1 => hailstoneSeq 3 = 3 : (10 : (0 : [1]))
returns            => 3 : 10 : 0 : [1]

Functions on lists

列表也可以使用模式匹配

-- Compute the length of a list of Integers.
intListLength :: [Integer] -> Integer
intListLength []     = 0
intListLength (x:xs) = 1 + intListLength xs

第一个子句匹配空列表并返回0，第二个子句匹配任何包含元素的列表，并用x指代第一个元素，用xs指代剩下的元素，然后返回1 + intListLength xs

既然我们没用到匹配到的x变量，不如用_替代它，习惯上_用于代表那些永远不会被用到的变量（但你又必须写一个占位符），所以可以改成

intListLength (_:xs) = 1 + intListLength xs

列表的模式匹配可以嵌套:

sumEveryTwo :: [Integer] -> [Integer]
sumEveryTwo []         = []     -- Do nothing to the empty list
sumEveryTwo (x:[])     = [x]    -- Do nothing to lists with a single element
sumEveryTwo (x:(y:zs)) = (x + y) : sumEveryTwo zs

最后一个子句匹配一个至少有两个元素的列表，用x, y分别指代第一、二个元素，并用zs指代剩下的元素，然后返回(x + y) : sumEveryTwo zs。嵌套时的括号不是必须的，因此sumEveryTwo (x:y:zs) = ...也是等价的

Combining functions

将复杂的函数分解为简单函数的组合是良好的Haskell编程风格

-- The number of hailstone steps needed to reach 1 from a starting
-- number.
hailstoneLen :: Integer -> Integer
hailstoneLen n = intListLength (hailstoneSeq n) - 1

这看起来可能很低效，为了计算hailstone需要的步数，居然要先生成一个完整的hailstoneSeq再计算长度，多浪费内存啊？但实际上并不！得益于Haskell的惰性求值，sequence的元素只有在被需要时生成，所以序列的生成和列表长度的计算是交织(interleaved)的。整个计算过程只使用O(1)的内存，不论输入多大的n（这里撒了一个善意的谎，但大差不差:p 至于问题在哪里就留待后几周讨论）

我们之后将会学习更多的关于Haskell的惰性求值策略。至于现在，需要了解的信息是: 不要担心写一个需要用到整个数据结构的函数，并尽量多写小函数来组合出复杂的功能，这一开始会不太自然，不过这是写理想的（并且高效的）Haskell程序的方式，并且当你习惯后，这样写代码是一种享受

A word about error messages

四字箴言: 别怕报错

GHC的报错信息可能看起来很长并且很吓人，但是它们长并不代表它们晦涩，而代表它们包含很多有用的信息，比如:

Prelude> 'x' ++ "foo"

<interactive>:1:1:
    Couldn't match expected type `[a0]' with actual type `Char'
    In the first argument of `(++)', namely 'x'
    In the expression: 'x' ++ "foo"
    In an equation for `it': it = 'x' ++ "foo"

报错第一行就告诉你 无法匹配传入的'Char'类型到期待的[a0]上 ，这意味着表达式中的某一部分被期待为一个[a0]（虽然不知道a0是什么，但是[]告诉你他是一个列表）但实际上却是Char，某一部分是哪一部分？第二行报错告诉我们它是(++)操作符的第一个参数，名字是'x'，下面的报错告诉我们这个表达式出现的位置

综合上述信息，我们很容易发现问题在于(++)操作符期待两个参数都是列表（进一步的，必须是相同类型的列表），而我们传入的第一个参数'x'是个Char

当你看到一大堆报错信息的时候，抵抗逃跑的冲动！深呼吸，然后仔细读错误信息。你不需要理解报错信息的全部内容，但是你肯定会学到很多，并且获得足够的定位问题的信息
