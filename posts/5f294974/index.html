Recursion patterns, polymorphism, and the Prelude//递归模板，多态与Prelude

在你完成作业2的时候，你可能花了很多时间写显式递归的函数，现在你可能认为那些就是Haskell程序员在大部分时间所做的。但实际上，有经验的Haskell程序员几乎从不写递归函数！

虽然在理论上，递归函数几乎能实现任何事（比如代替循环来迭代列表），不过在实践中我们会发现，有一些模式在递归中反复出现，通过抽象这些模板到库函数，我们就能把低级的、递归的逻辑剥离到库函数中，而只需要实现高阶的逻辑，这也是wholemeal programming的目的

Recursion patterns//递归模板

回忆我们对Int类型的列表的简单实现:

data IntList = Empty | Cons Int IntList
  deriving Show

我们可能会希望在IntList上实现什么操作呢？以下是几个可能的选择:

  • 遍历列表，对每个元素做操作（映射）
  • 仅保留一部分符合指定要求的元素（过滤）
  • 总结整个列表的性质，比如求所有元素的和、乘积、最大值最小值...（规约）
  • ...

译者注: 如果你学过Javascript或Python等，并对Map/Filter/Reduce等高阶函数十分了解，那么下面的这部分内容可以快速浏览

Map//映射

我们先考虑第一个操作: 遍历列表并对每个元素做操作，比如我们可以对每个元素加一:

incAll :: IntList -> IntList
incAll Empty = Empty
incAll (Cons x xs) = Cons (x+1) (incAll xs)

也可以对每个元素求绝对值:

absAll :: IntList -> IntList
absAll Empty = Empty
absAll (Cons x xs) = Cons (abs x) (absAll xs)

也可以对每个元素求平方

squareAll :: IntList -> IntList
squareAll Empty       = Empty
squareAll (Cons x xs) = Cons (x*x) (squareAll xs)

此时此刻，你的脑子里该响防空警报了。这三个函数长得实在太像了，我们是不是在repeating outselves？这可犯了大忌。为了避免重复，我们必须抽象共性

上面的三个函数，所做的都是一件事情: 对列表的每一个元素应用一个操作(inc, abs, square)，然后返回由新元素组成的列表。这种操作即为Map，如果你学过Javascript或者Python，那对此(以及剩下的所有操作)肯定不陌生

在每一个Map操作中，遍历每个元素并应用操作这一点是不变的，而每次应用的操作具体是什么是可变的，但他们都是一个Int -> Int的函数——而在Haskell中，函数是一等公民，因此，要做的操作本身可以被作为参数，指示负责遍历的函数究竟要做什么操作:

mapIntList :: (Int -> Int) -> IntList -> IntList
mapIntList _ Empty        = Empty
mapIntList op (Cons x xs) = Cons (op x) (mapIntList op xs)

mapIntList的骨架几乎和我们之前实现的函数一模一样: 空边界判断、递归遍历每一个元素并应用操作... 唯一的区别在于操作本身由参数指定——它是一个Int -> Int的函数

现在我们可以改写之前的三个函数了:

exampleList = Cons (-1) (Cons 2 (Cons (-6) Empty))

addOne :: Int -> Int
addOne x = x + 1

square :: Int -> Int
square x = x * x

mapIntList addOne exampleList    -- same as incAll
mapIntList abs    exampleList    -- same as absAll
mapIntList square exampleList    -- same as squareAll

Filter//筛选

第二个希望的操作是，仅保留一部分符合指定要求的元素，即筛选我们想要的元素，比如下面这个只保留偶数的函数:

keepOnlyEven :: IntList -> IntList
keepOnlyEven Empty = Empty
KeepOnlyEven (Cons x xs)
	| even x    = Cons x (keepOnlyEven xs)   -- is even, keep it
	| otherwise = keepOnlyEven xs            -- drop it away

有了Map的经验，很容易能想到，我们依然可以实现一个遍历列表的骨架，而究竟哪个元素需要保留，则可以由我们传递一个测试函数来实现，测试函数则是一个接受Int，返回是否保留它的信息(Bool)的函数:

filterIntList :: (Int -> Bool) -> IntList -> IntList
filterIntList _ Empty = Empty
filterIntList test (Cons x xs)
	| test x    = Cons x (filterIntList test xs)
	| otherwise = filterIntList test xs
	
filterIntList even exampleList

Fold//折叠

最后一个希望的操作是，总结整个列表的某个属性，并返回一个单一值，比如求所有元素的和、乘积、最大值最小值...这种操作在Haskell中被称为Fold，而在其他地方可能也被叫做Reduce，即规约。参照上面两个例子，你可以自己想一下这个操作该如何抽象，并自行实现一个reduceIntList，关于它的讨论我们会在下周展开

我的一个参照实现:

-- 这是一个实现不完全的fold，因为我们只能获得Int类型的"总结"，想知道如何支持更多类型，接着学吧
foldIntList :: Int -> (Int -> Int) -> IntList -> IntList
foldIntList initVal _ Empty = initVal
foldIntList currVal op (Cons x xs) = foldIntList (op x currVal) op xs

Polymorphism//多态

我们现在已经写了一点漂亮又通用的函数来帮助完成在IntList上的map或者filter的操作了，但是这还不够——如果我们如果我们要操作的不是IntList呢，我们难道还要分别实现mapStringList, mapIntegerList, mapBoolList, mapListList, ...吗，那就又是在重复了——这所有函数的逻辑都一模一样，只有里面的类型标记不同——我们最忌讳的就是重复

当然不能对每一个数据类型都实现一个一样功能的函数，既然只有数据类型不同，那么就想Haskell有没有能支持我们对一个逻辑复用不同数据类型的解决办法呢？多态(polymorphic)就是

polymorphic 源自希腊语(πολύμορφος)，意为“拥有多种形式”

Polymorphic data types

首先，我们来看看如何定义一个多态数据类型

data List t = E | C t (List t)

因为之前已经在IntList里使用了Empty和Cons两个名字来指代我们的两个数据构造器，所以现在不直接使用这两个名字以避免混淆，在这段定义里，E代表Empty，C代表Cons

其中多出来的t是一个类型变量(type variable)，他可以指代任何类型，比如Int。语法上，类型变量必须以小写字母开头，类型必须以大写字母开头

在增加了t 之后，我们可以给List这个类型构造器传递一个Type作为参数，比如Int，得到一个特化的、指定了元素类型的列表类:

lst1 :: List Int
lst1 = C 3 (C 5 (C 2 E))

lst2 :: List Char
lst2 = C 'x' (C 'y' (C 'z' E))

lst3 :: List Bool
lst3 = C True (C False E)

我们不必再分别实现IntList, CharList, BoolList...了，只要实现一次List t，然后传递我们想要的目标类型作为构造参数就好

Polymorphic functions

现在，来实现我们的最终目标: 依赖于多态类List让我们之前实现的函数也多态化。先从filterIntList开始下手:

-- Old version
filterIntList :: (Int -> Bool) -> IntList -> IntList
filterIntList _ Empty = Empty
filterIntList test (Cons x xs)
	| test x    = Cons x (filterIntList test xs)
	| otherwise = filterIntList test xs

-- Brand new, polymorphic version
filterList :: ??? -- see below
filterList _ E = E
filterList test (C x xs)
  | test x    = C x (filterList test xs)
  | otherwise = filterList test xs

在ghci里输出filterList的类型，我们得到的是

*Main> :t filterList
filterList :: (t -> Bool) -> List t -> List t

这可以读作: 对于类型t，filterList接受一个映射t到Bool的函数(即test函数)和一个由许多类型为t的元素构成的列表，然后返回一个这样的列表

那么mapIntList的泛化版本呢，mapList函数的类型定义应该是什么？依样画葫芦，参照filterList的类型，我们可能第一时间会想到:

mapList :: (t -> t) -> List t -> List t

除了接受的函数从返回Bool变成返回t以外，没什么改变。这个类型定义是可用的，你可以自己实现一遍来验证。不过这种写法还是差了一点: 它可以支持把一个由t类型构成的列表映射成一个t类型构成的列表，也仅能如此，对于把每个元素加一、取取绝对值这样的需求，它能胜任，但假如我们做mapList show这样的调用，来实现展示整个列表里所有元素(的字符串形式)，这个函数就做不到了。根本原因在于它要求输入的列表元素类型和输出的都是t，如果打破这个要求呢:

mapList :: (a -> b) -> List a -> List b
mapList _ E        = E
mapList f (C x xs) = C (f x) (mapList f xs)

这是我们新实现的map，现在它的定义中出现了两个不确定的类型，一个决定了输入的列表的元素类型，另一个决定输出的。多态函数在调用时决定类型，因此对任何可能的类型，我们的mapList函数都能正常工作了(当然也包括a,b相同的情况，这种情况下，新的mapList退化成旧的)

The Prelude

Prelude是一个包含一组基础定义的模组(module)，它会被所有Haskell程序隐式地引用，翻阅它的文档来了解它究竟能干什么是很值得的事情。在Prelude中已经定义了多态列表，还实现了许多有用的多态函数，比如map, filter，对应着我们自己实现的mapList, filterList，其中Data.List模块实现了更多这些函数

译注: 原文指向的Prelude文档(http://haskell.org/ghc/docs/latest/html/libraries/base/Prelude.html)已经404，现在文中的链接都是我找到的版本

其中一个有用的多态类型叫做Maybe，它是这样定义的:

data Maybe a = Nothing | Just a

一个类型为Maybe a的变量要么真的包含了一个a(被Just构造器包裹)，要么就是Nothing(通常用来表示错误)，Data.Maybe模块包含了针对Maybe的实用函数

Total and partial functions//全函数与部分函数

考虑这样一个多态类型

[a] -> a

什么样的函数会有这样的类型呢？这个类型表明给定一个由a类元素构成的列表，函数返回一个a类型的值，比如Prelude中的head干的就是这件事: 给定任何类型的列表，head返回它的第一个元素

但是如果head拿到了一个空列表呢？我们来看看它的源码...

-- | \(\mathcal{O}(1)\). Extract the first element of a list, which must be non-empty.
--
-- >>> head [1, 2, 3]
-- 1
-- >>> head [1..]
-- 1
-- >>> head []
-- Exception: Prelude.head: empty list
head                    :: [a] -> a
head (x:_)              =  x
head []                 =  badHead
{-# NOINLINE [1] head #-}

badHead :: a
badHead = errorEmptyList "head"

如你所见，head在遇到空列表时的做法是抛出异常，因为除此以外也没有合理的处理办法——head甚至不知道这里的类型a是什么，它可能是任何类型，没人能凭空变出这么一个元素

head就是一个部分函数(partial function): 有一些输入会导致head崩溃。那些在某些case下递归不会终止的函数也叫部分函数。那些定义良好、在所有可能的case下都能良好工作的函数叫全函数(total functions)

良好的Haskell实践意味着尽可能避免部分函数——这也是在任何编程语言中的良好实践——可惜在大部分语言中，要实现避免部分函数非常非常困难，Haskell希望让这件事尽量简单

head就是个错误！它不应该出现在Prelude中，其他的Prelude中的部分函数还包含tail, init, last, (!!)，你永远都不应该使用这些函数，你要是在作业里使用这些函数，我就狠狠地扣你的分XD

那么，应该如何找到它们的替代品呢？

Replacing partial functions//替代部分函数

通常，像head,tail这种部分函数可以用模式匹配来替代，比如下面这两个例子:

doStuff1 :: [Int] -> Int
doStuff1 []  = 0
doStuff1 [_] = 0
doStuff1 xs  = head xs + (head (tail xs)) 

doStuff2 :: [Int] -> Int
doStuff2 []        = 0
doStuff2 [_]       = 0
doStuff2 (x1:x2:_) = x1 + x2

它们做的事一样的事，返回一样的结果，并且都是全函数，不过只有第二个才是一眼就能看出是全函数的，第一个则难读很多。记得多写像第二个一样好读的代码

Writing partial functions//写部分函数

假如你发现自己在写部分函数，该怎么办？给你两条可选的路，第一是修改函数的输出类型来指示它可能会遇到的错误，回想一下Maybe的定义:

data Maybe a = Nothing | Just a

现在假设我们正在实现head，我们可以把它改写成这个安全的版本:

safeHead :: [a] -> Maybe a
safeHead []    = Nothing
safeHead (x:_) = Just x

这也是在safe package里真实存在的函数，这样设计的理由有这些:

  1. safeHead永远不会崩溃
  2. safeHead的类型定义让它发生的错误非常显而易见，可以直接按返回值是否是Nothing判断
  3. 类型系统会强迫调用safeHead的人正确实现处理返回Nothing情况的逻辑，否则编译不会通过（一举打倒了许多偷懒的邋遢鬼

在一些场景下，safeHead仍然是“部分的”，不过我们已经在类型系统中反映了这个“部分性”，至少在这种意义上它是安全的。我们已经让函数的类型尽可能多地透露关于它是如何工作的信息了

不过假如我们保证只会对非空的列表调用head呢？在这种情况下，收到Maybe a作为返回值可能会比a麻烦，因为我们还要花功夫去处理永远不可能出现的返回值是Nothing的情况

解决办法，也是我给你的第二条路是，如果一些情况确实担保了边界条件不可能触发，那么我们的类型应该能反映出这个担保，来让编译器确认这个担保是真的实现了，而不是忽悠人的花招:

data NonEmptyList a = NEL a [a]

nelToList :: NonEmptyList a -> [a]
nelToList (NEL x xs) = x:xs

listToNel :: [a] -> Maybe (NonEmptyList a)
listToNel []     = Nothing
listToNel (x:xs) = Just $ NEL x xs

headNEL :: NonEmptyList a -> a
headNEL (NEL a _) = a

tailNEL :: NonEmptyList a -> [a]
tailNEL (NEL _ as) = as

你可能觉得做这么多麻烦的工作，只是为了帮助那些不像你一样天才的笨蛋，因为你永远不会做出对空列表调用head这种蠢事，对吧？世上肯定有这样的笨蛋，不过他们可能不是你想的那些人;)
