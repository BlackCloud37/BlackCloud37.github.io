Rust Quiz #1

Answer: 112

  • 在宏里，$(...)*代表接受0或多个被包裹的Designators(相应的，$(...)+则是一个或多个)，也可以在其中插入分隔符的pattern，以下是一些例子

    ($($s:stmt)*)           matches ({} {} {})   // {} is stmt
($x:expr, $($y:expr)+)  matches (1, 1 1)     // the last two `1` matches $y twice
($x:expr, $($y:expr),+) matches (1, 1, 1)    // almost same as previous one, but has `,` in pattern

    因此，题中的宏能匹配多个statement，它们之间不需要分隔符

  • $(...)<Token>*语法用于展开上条中通配符收集的Designators，以题目中的宏为例，展开的例子如下(来自quiz的官方解答)

    Suppose we call this macro with three of the statements shown above as input.

    m! {
    struct S { x: u64 }
    let mut s = S { x: 1 }
    s.x + 1
}

    The macro expands to:

    { stringify!(struct S { x: u64 }); 1 }
    << { stringify!(let mut s = S { x: 1 }); 1 }
    << { stringify!(s.x + 1); 1 }

    Each of the stringifys expands to a string literal:

    { "struct S { x: u64 }"; 1 }
    << { "let mut s = S { x: 1 }"; 1 }
    << { "s.x + 1"; 1 }

  • 由于展开后的每个block stmt都是一个返回1的expr，因此上面的展开结果等价于1 << 1 << 1，即2^(3-1)=4，即该宏会返回2^(n-1)，n是调用宏时匹配到的语句个数，因此需要考虑题目中三处调用的语句数目

  • 考虑题目中给的三处宏调用

    • return || true 是单条语句，语义为返回一个函数closure: || true
    • (return) || true 是单条语句，语义为(return) logic-or true，(return)是一个表达式
    • {return} || true 是两条语句{return}和|| true，因为前者是单独的块语句

  • 因此答案为112

Rust Quiz #5

Answer: 112

注：并不很确定下面的解释是否正确，因为没太理解官方的解释，也没找到其他题解

此处为两种函数指针实现了Trait

  • fn(T)
  • for<'a> fn(&'a T)

而在main中，使用三个函数指针指向f, g两个函数，编译器会自动推断其中的_为对应类型，并根据推断结果选择其Trait的表现

  • let a: fn(_) = f; 由于f(_: u8)，此处_ = u8，因此a: fn(u8)适用第一个Trait
  • let b: fn(_) = g; 由于g(_: &'x u8)，此处_ = &'x u8，生命周期参数'x确定推断自g，因此&'x u8即对应了一个具体类型T，适用第一个Trait
  • let c: fn(&'a _) = g; 此处_ = u8，此处的'a需要推断自c，c本身也就构成了一个高阶闭包类型，因此c: for<a'> fn(&'a u8)，适用于第二个Trait

Rust Quiz #7

Answer: 1

此处的枚举确实等价于

#[repr(u8)]
enum Enum {
    First = 0u8,
    Second = 1u8,
}

所以transmute(1u8)对应的也确实是Enum::Second，但结果仍然是1，因为match块的写法

match self {
    First => print!("1"),
    Second => print!("2"),
}

这里的First并不是Enum::First这个pattern，而是接受任意pattern并将其绑定到名为First的变量上，所以第一个match arm永远会被选择

Rust Quiz #8

Answer: 1214

宏在处理pattern时会将其中符合Rust token的符号组合到一起，然后施行匹配，在组合过程中从左到右、从长到短匹配符合的token，比如下面这几个pattern会被处理为

<<= 对应 <<=   // <<=本身就是一个token，代表左移赋值
=<< 对应 = <<  // 从左到右，从长到短，依次检查=<<, =<, =，发现=是token，拿走，然后对剩余部分继续

==>   对应 == >
= = > 对应 = = >
== >  对应 == >
= =>  对应 = =>

回头看macro_rules，可以知道它们匹配的pattern实际是

macro_rules! m {
    (== >) => { print!("1"); };
    (= = >) => { print!("2"); };
    (== >) => { print!("3"); };
    (= =>) => { print!("4"); };
}

而在将一个pattern传递给m!的时候，也会做上述组合过程，因此第三个pattern arm永远不会被执行，因为它的pattern和第一条一样

Rust Quiz #9

Answer: 21

从t!开始展开宏

t!(1) 
=> { e!(1); m!(1); } // 两个 1 都是 token tree
=> { m!($e); m!(1); } // $e是expr，它会被evaluate成1，但expr对于macro是不可见(opaque)的，即在其被执行之前，程序都不知道这个表达式是1，因此也不会被匹配到 m! 的第一个arm

以下宏参数都是不可见(opaque)的

  • $:block
  • $:expr
  • $:item
  • $:literal
  • $:meta
  • $:pat
  • $:path
  • $:stmt
  • $:ty

而这些是可见的

  • $:ident
  • $:lifetime
  • $:tt

Rust Quiz #14

Answer: 10

trait的impl在任何位置都会在整个作用域上生效(always in scope)，因此char的Trait也在作用域中生效，'?'.is_reference()会输出0

而0.is_reference()会输出1是因为Rust的auto-ref规则，参考What are Rust's exact auto-dereferencing rules?，在试图调用is_reference时，先从调用者类型int开始找是否有相应的trait提供了此方法，但并没有找到(int并没有实现Trait)，然后开始auto-ref，结果是在&0类型上发现了is_referencing方法，并最终调用(&0).is_reference()，输出1

Rust Quiz #15

Answer: 1

对x做类型推断，可知x: &{integer}，integer的具体类型需要在x.f()推断

试图在x上找到f方法，并借此确定integer的类型

  • 假设推断integer: i32，即x: &'a i32，此时x被实现了Trait，但其上并没有以self为caller的f方法，因此x.f不存在
  • 假设推断integer: u32，则x: &u32，此时由于u32被实现了Trait，且其上有&self为caller的f方法，因此满足

所以x.f()最终调用u32的Trait的方法

Rust Quiz #19

Answer: 21

let _ = s;可以理解为什么都没干（这取决于_在Rust中的语义，可以看rust-lang/rust#10488），没有所有权转移，也没有s的复制，什么都没发生，因此程序先输出2，然后s out of scope被销毁，输出1

  • 如果把_替换成一个普通的变量名，如a，那么s的所有权就会在let a = s;处转移（因为我们没有为S提供Copy trait），不过此时仍然只有一个S，只不过其所有权被交给了a，程序依然会输出21
  • 如果把let _ = s;替换成s = S这种move without binding的语句，则前一个S会在此处立刻被drop

Rust Quiz #20

Answer: 121

考虑四个函数的语义

  • fn return1

    if的条件是(return { print!("1") })这样一个表达式，该表达式返回()，且在求表达式结果的过程中print!被执行

  • fn return2

    同return1，return的优先级高，可以把这个高优先级理解为包一层括号

  • fn break1

    由于有括号，括号内的语句被解释为break-with-value expression，因此为了break，要先求其value，进而执行print!

  • fn break2

    由于没有括号，此处的break语句本身被作为if的条件，因此在print!前循环就已退出

Rust Quiz #21

Answer: 221111

// re-write the code verbosely
let x = || { 
  (return) || true;
};
x().f();

// x() = ()
let x = loop { 
  (break) || true; 
};
x.f();

// loop will break with return value ()
// so x = ()
let x = || { 
  return (|| true); 
};
x().f();

// x() = (|| true), a closure
let x = loop { 
  break (|| true); 
};
x.f();

// x = (|| true)
let x = || { 
  return || true; 
};
x().f();

// x() = || true
let x = loop { 
  break || true; 
};
x.f();

// x = || true

Rust Quiz #27

Answer: 11

  • Supertraits are not inheritance. Supertraits are a constraint that if some trait is implemented, some other trait must also be implemented. 因此Base和Derived的method没什么关系

  • 对dyn Base调用方法最终会被转发到Base Trait的相应方法

    // Generated by the compiler.
//
// This is an implementation of the trait `Base` for the
// trait object type `dyn Base`, which you can think of as
// a struct containing function pointers.
impl Base for (dyn Base) {
    fn method(&self) {
        /*
        Some automatically generated implementation detail
        that ends up calling the right type's impl of the
        trait method Base::method.
        */
    }
}

  • 通过泛型特化T: Base会让编译器生成一份fn static_dispatch (x: Base) {...}版本的实现，因此此处也会调用Base的方法

Answer: 1244

  • (0)只能被推断为u32因为只有这样才有可调用的p
  • (0,)只能被推断为(i32,)，同上
