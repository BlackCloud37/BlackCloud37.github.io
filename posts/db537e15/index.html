More polymorphism and type classes//更多多态与Type class

Haskell多态系统的特色是参数多态 (parametric polymorphism)，即多态函数可以处理任何输入类型，这对于使用多态的程序员或多态函数的使用者都有有趣的影响

Parametricity

考虑这个类型

a -> a -> a

这里的a是一个可以指代任何类型的类型变量，什么函数有上面这样的类型呢？

这样的？

f :: a -> a -> a
f x y = x && y

但这个函数没法通过编译，虽然它的语法看上去没问题，但是它无法通过类型检查(type check)，编译器会报这个错:

2012-02-09.lhs:37:16:
    Couldn't match type `a' with `Bool'
      `a' is a rigid type variable bound by
          the type signature for f :: a -> a -> a at 2012-02-09.lhs:37:3
    In the second argument of `(&&)', namely `y'
    In the expression: x && y
    In an equation for `f': f x y = x && y

编译错误的理由是，多态函数的调用者才是决定其参数类型的人，但在这里，函数的实现者却先假定了a是Bool，并对其使用了Bool专属的&&运算符。要使用这个函数的人不一定事先知道我们用了&&，因此他可能试图用String或者Int作为参数来调用这个函数，然后发现出了问题——这显然是个不合理的情况，因此编译器选择拒绝这种实现。换句话说，你可以把a -> a -> a这个类型看成是一个承诺(promise)，承诺这个函数可以接受调用者给的任何类型的参数

另一个我们可以想到的实现方式是

f a1 a2 = case (typeOf a1) of
            Int  -> a1 + a2
            Bool -> a1 && a2
            _    -> a1

f函数对不同的类型有不同的表现，在Java里，我们确实可以这样实现它:

class AdHoc {

    public static Object f(Object a1, Object a2) {
        if (a1 instanceof Integer && a2 instanceof Integer) {
            return (Integer)a1 + (Integer)a2;
        } else if (a1 instanceof Boolean && a2 instanceof Boolean) {
            return (Boolean)a1 && (Boolean)a2;
        } else {
            return a1;
        }
    }

    public static void main (String[] args) {
        System.out.println(f(1,3));
        System.out.println(f(true, false));
        System.out.println(f("hello", "there"));
    }

}

[byorgey@LVN513-9:~/tmp]$ javac Adhoc.java && java AdHoc
4
false
hello

不过在Haskell里，这样写是不被允许的——因为它没有类似Java的instanceof的操作符，它也没有能力在运行时查询一个变量的类型，并根据变量的类型决定具体要做什么。这种能力的缺失的一个理由是，Haskell的类型在类型检查完成后就被编译器抹去了: 在运行时，根本没有可以用来查询的类型信息。不过，我们还有其他更好的理由

我们提过这种多态的实现风格被称为参数多态，像f :: a -> a -> a这样的函数对于类型a是parametric的，“parametric”的含义是“对任何调用者选择的参数类型都能工作”。在Java中，这种多态能力是由泛型提供的

泛型就是受到Haskell启发而产生的，Haskell的一个主要设计者Phili Wadler就是后来Java泛型系统的开发者

所以什么函数实现才能有a -> a -> a这样的类型呢，以下是两个例子:

f1 :: a -> a -> a
f1 x y = x

f2 :: a -> a -> a
f2 x y = y

由此可见， a -> a -> a这个类型透露了很多关于函数的信息

现在，看看以下几个类型，思考一下什么样的函数可能具有这些类型

  • a -> a
  • a -> b
  • a -> b -> a
  • [a] -> [a]
  • (b -> c) -> (a -> b) -> (a -> c)
  • (a -> a) -> a -> a

Two views on parametricity

作为一个多态函数的编写者，特别是当你之前经常用instanceof之类的语法来实现多态函数时，你现在肯定觉得Haskell的诸多限制非常不便: “怎么连这都没法用？” 但从多态函数使用者的视角来看，参数并不是“调用该函数的限制(restrictions)”，而是“对于该函数功能的保证(guarantees)”。当函数通过各种途径提供了对其功能的有力保证时，推理工具的使用就容易多了

不过，确实有些时候根据变量类型决定做什么操作非常有用，比如加法，我们之前已经知道加法是多态的，Int, Integer和Double都可以做加法，显然它必须知道被加的数是什么类型: Integer的加法和Double的加法是完全不同的操作。它是怎么实现根据加数的类型决定具体的加法操作的呢？

我们不妨看看(+)的类型:

Prelude> :t (+)
(+) :: Num a => a -> a -> a

Emmm, Num a =>是什么意思？实际上，(+)不是唯一一个在定义里包含=>的基础函数，下面这些函数的定义里也有一样的符号:

(==) :: Eq a   => a -> a -> Bool
(<)  :: Ord a  => a -> a -> Bool
show :: Show a => a -> String

下面我们来看看它们的实际意义

Type classes//类型类

Num, Eq, Ord, 和 Show都叫类型类(type classes)，而(==),(<),(+)这些类型定义里包含类型类的函数都是类型类多态的(type-class polymorphic)。类型类指的是一些类型的集合，每个集合中的所有类型都支持共有的操作，针对某个类型类的 type-class polymorphic 函数则只对该集合的成员类型有效，举个例子，有一个类型类叫做Eq:

class Eq a where
  (==) :: a -> a -> Bool
  (/=) :: a -> a -> Bool

我们可以这样解释这段定义: Eq是一个包含单参数a的类型类，任何类型a想作为Eq类的集合中的一员，就必须支持两个函数(==)和(/=)，并且这两个函数必须符合对应的函数定义。比如如果我们想让Int成为Eq的一员，就要定义两个函数(==) : Int -> Int -> Bool和(/=) :: Int -> Int -> Bool。因为Haskell已经实现了针对Int的这两个函数，因此实际上Int已经是Eq的成员类了

现在我们再回过头来看(==)的定义:

(==) :: Eq a => a -> a -> Bool

=>前的Eq a是该定义的类型类约束(type class constraint)，可以理解为只要类型a是Eq的成员，(==)函数就能接受两个a类型的变量并返回一个Bool，而对非Eq成员的类型调用(==)则会导致类型错误。如果说一个普通的多态函数保证了函数能正确处理任何类型，类型类多态的函数则保证能正确处理符合其类型类约束的类型

Haskell编译器会对类型类函数做类型推断，以根据其被调用时的参数的具体类型确定究竟要使用哪个版本的该函数，其表现就像Java/C++中的重载一样

为了加深理解，我们再来自己实现一个属于Eq的类型:

data Foo = F Int | G Char

instance Eq Foo where
  (F i1) == (F i2) = i1 == i2
  (G c1) == (G c2) = c1 == c2
  _ == _ = False

  foo1 /= foo2 = not (foo1 == foo2)

如上，我们实现了一个类型Foo，并且告诉编译器它是一个Eq的成员: 我们帮它实现了(==)和(/=)两个函数。不过如果每次都要为了让一个类属于某个类型类而实现一遍这些函数可能会很麻烦，而实际上确实有一个捷径: 类型类的那些共有函数通常可以基于某些基础函数实现，只要我们不覆盖其他函数的定义，那些函数就会默认按照基础函数的实现工作。举个例子，还记得Eq的类型声明如下:

class Eq a where
  (==) :: a -> a -> Bool
  (/=) :: a -> a -> Bool
  x /= y = not (x == y)

那么之后所有被声明为Eq成员的类都只需要实现(==)函数，然后它们的(/=)函数就会自动采用not ==的结果，而不必手动实现一遍。不过如果你希望/=有一些特殊表现，你也可以手动实现一遍，用你的实现覆盖默认的not ==

Haskell的Eq类型类的真实定义如下:

class Eq a where
  (==), (/=) :: a -> a -> Bool
  x == y = not (x /= y)
  x /= y = not (x == y)

这意味着要声明一个类为Eq的成员，我们可以选择实现(==)或者(/=)中的任何一个，另一个就会被自动实现为前一个的not——我们可以觉得实现哪个方便就写哪个，更方便了。不过如果两个都没实现，这里就会无限递归，(==)需要依赖(/=)，(/=)又要求(==)先实现...

更方便的是，GHC会自动帮我们实现一些基础类型类的实例，比如要声明我们的Foo'为Eq, Ord, Shoq这三个基础类型类的实例，只需要这样写:

data Foo' = F' Int | G' Char
  deriving (Eq, Ord, Show)

然后GHC就会自动帮我们把Foo'变成这三个类型类的实例了

Type classes and Java interfaces//类型类和Java interface

类型类和Java的interface十分相似，它们都定义了一类类型(type/class)，这类类型都必须实现特定的操作，不过类型类比Java interface更通用，这体现在以下几点上:

  1. 当Java的类被定义时，任何其实现的interface都必须被声明。而类型类的实例则可以在不同的地方被声明，甚至可以是在不同的模块中

  2. 类型类函数的声明方式比interface函数的签名更灵活且通用，特别是对于多参数函数，比如对于下面这个类型类例子:

class Blerg a b where
  blerg :: a -> b -> Bool

Blerg类型类的blerg方法的选用同时依赖于a和b两个类型，而Java中没有类似的能力

Haskell的类型类还能轻松驾驭二元（甚至三元，四元...）的函数，比如

class Num a where
  (+) :: a -> a -> a
  ...

在Java中没有实现这种效果的好办法: 两个参数中的一个必须是“特权”参数，这个参数实际被用来调用(+)方法，而这种不对称性非常尴尬；此外，由于Java有子类这一概念，两个同样类型的参数并不实际是同一类型（而可能只是继承自同一父类），这使得实现诸如(+)之类的二元运算符需要做运行时的类型检查

Standard type classes//标准类型类

除了Eq外，你还需要了解Haskell中的其他标准类型类:

  • Ord代表着那些可以被排序的类型，即属于Ord的类型都必须实现诸如(<), (<=)和compare等用于比较的函数

  • Num代表着那些数值类型，它们需要实现加法、减法和乘法操作。需要注意的是Haskell的整数字面量通常是类型多态的(函数):

    Prelude> :t 5
5 :: Num a => a

    即像5这样的字面量的类型可以是Int/Integer/Double或者是任何其他Num的成员类

  • Show代表着那些可以被打印的类，它们必须实现show函数，用于把该类型转化为可输出的String

  • Read则是Show的反面，它们必须实现可以将String构造成其类型实例的函数

  • Integral则代表像Int和Integer这样的整型

A type class example//一个类型类的例子

下面是一个我们自己实现的类型类的例子:

class Listable a where
  toList :: a -> [Int]

我们可以把Listable看作是一类可以被转化成一个Int列表的类型，它需要实现toList函数:

toList :: Listable a => a -> [Int]

然后我们可以定义一些Listable的实例了，首先，一个Int肯定能被直接转化成[Int]，只要把它自己作为该列表的唯一元素就好；而Bool可以一样转换，只要先把True变成1，False变成0:

instance Listable Int where
  -- toList :: Int -> [Int]
  toList x = [x]

instance Listable Bool where
  toList True  = [1]
  toList False = [0]

[Int]本身不需要任何转化就能变成[Int]，所以它也是Listable的实例:

instance Listable [Int] where
  toList = id

一个二叉树也可以被拍平成一个列表:

data Tree a = Empty | Node a (Tree a) (Tree a)

instance Listable (Tree Int) where
  toList Empty        = []
  toList (Node x l r) = toList l ++ [x] ++ toList r

如果我们用toList来实现其他函数，那么它们也会包含Listable这一类型约束，比如对于:

-- to compute sumL, first convert to a list of Ints, then sum
sumL x = sum (toList x)

ghci会告诉我们sumL的类型是:

sumL :: Listable a => a -> Int

即sumL只能对Listable的成员类生效，因为它的实现依赖了toList，而toList只对Listable成员类起效。再看看下面这个函数:

foo x y = sum (toList x) == sum (toList y) || x < y

ghci告诉我们foo的类型是:

foo :: (Listable a, Ord a) => a -> a -> Bool

即foo只能对既是Listable又是Ord的类型起作用，因为它同时用到了toList和比较函数

最后也是最复杂的例子:

instance (Listable a, Listable b) => Listable (a,b) where
  toList (x,y) = toList x ++ toList y

看一看我们如何把实例和函数类型声明为Listable的成员，这段定义的意思是当一个pair类型包含的两个类a,b都是Listable时，这个pair类型(a,b)也是Listable，且它的toList的结果基于a和b的toList的拼接
